# Binance-Hyperliquid 共同永续对改造 TDD 待办清单

## 1. 文档目标

本清单用于指导当前交付版从“演示型监控”升级到“先发现 Binance 与 Hyperliquid 共同永续交易对，再按价差或费率差排序展示”的目标实现，并以 TDD 模式拆分开发顺序。

适用目标：

- 后台先发现两所共同永续交易对。
- 实时价差页基于共同交易对订阅与计算。
- Funding 页基于共同交易对动态生成机会，而不是依赖静态快照。
- 前端支持按价差、费率差或净收益等维度排序展示。

## 2. TDD 原则

### 2.1 开发原则

- 先写失败测试，再实现最小代码，再重构。
- 先覆盖纯函数与状态逻辑，再补页面渲染与交互测试。
- 每完成一个阶段都要补齐回归测试，避免破坏现有功能。
- 交集计算、排序规则、异常降级必须优先做成可独立测试的纯逻辑模块。

### 2.2 测试分层

- 单元测试：覆盖市场标准化、交集计算、价差计算、funding 计算、排序规则、状态转换。
- DOM 测试：覆盖排序切换、共同交易对数量展示、空态与异常态展示、交集刷新后的 UI 联动。
- E2E 冒烟测试：覆盖市场发现、进入价差页、进入 funding 页、切换排序、异常降级等主路径。

### 2.3 推荐组织方式

- `core`：纯函数，如 symbol 标准化、交集计算、spread 计算、funding 计算。
- `services`：市场清单拉取、缓存、订阅编排。
- `state`：共同交易对、行情状态、排序状态、错误状态。
- `render`：列表展示、状态标签、排序控件、空态文案。

## 3. 当前实现与目标实现对照

| 维度 | 当前交付版现状 | 目标实现 |
| --- | --- | --- |
| 市场发现 | 未做全市场扫描 | 分别拉取 Binance / Hyperliquid 永续市场清单 |
| 共同交易对求交集 | 无独立交集模块 | 增加标准化与交集计算模块 |
| 价差页标的来源 | 写死 `DEFAULT_SYMBOLS` | 改为交集结果驱动订阅 |
| 价差参与条件 | 某标的两边都有盘口才计算 | 先基于交集订阅，再对双边就绪标的排序 |
| 价差排序 | 按净价差绝对值排序 | 支持按净价差、毛价差、成交量、综合评分排序 |
| Funding 数据来源 | `mockData` 静态快照 | 改为双边实时或准实时数据动态生成 |
| Funding 排序 | `estimatedNetHourly` 优先，`fundingSpreadHourly` 次级 | 支持按费率差、净收益或综合评分排序 |
| 展示层职责 | 同时承担演示和部分计算逻辑 | 前端专注展示，后台负责发现、归一化、计算 |
| 扩展性 | 更偏 MVP 演示 | 可扩展到多交易所、多排序模式、多告警 |

## 4. 目标流程

1. 拉取 Binance 永续市场清单。
2. 拉取 Hyperliquid 永续市场清单。
3. 做交易对命名标准化和映射。
4. 求出两所共同永续交易对。
5. 基于交集建立盘口与 funding 订阅。
6. 为每个共同标的动态生成价差机会与 funding 机会。
7. 按用户所选排序规则输出列表。
8. 在前端展示机会、状态、排序依据和异常提示。
