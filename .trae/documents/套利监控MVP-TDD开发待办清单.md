# 套利监控 MVP TDD 开发待办清单

## 1. 文档目标

基于以下两份文档整理 MVP 对应的开发待办清单，并以 TDD 思路拆分执行顺序：

- `套利监控MVP-产品需求文档.md`
- `套利监控MVP-技术架构文档.md`

本文档聚焦当前阶段的 MVP 范围：

- 仅交付单文件 HTML 原型。
- 使用静态示例数据与交互演示。
- 不接入真实交易接口。
- 为未来接入 `Python + ccxt + Grafana + VictoriaMetrics + SQLite` 预留结构。

## 2. 开发原则

### 2.1 TDD 原则

- 先写失败测试，再实现最小功能代码。
- 先覆盖纯函数与状态逻辑，再覆盖 DOM 渲染与页面交互。
- 每完成一个模块，都要补齐回归测试，避免后续迭代破坏已有行为。

### 2.2 代码风格原则

- 数据、状态、渲染、事件编排分层组织。
- 纯函数优先，避免在 DOM 事件回调中堆积业务逻辑。
- 保持命名清晰、职责单一、模块边界稳定。
- 开发态允许模块化拆分，交付态再汇总为单文件 HTML。

### 2.3 推荐实现方式

- 开发结构建议拆分为 `state`、`core`、`render`、`fixtures`、`styles` 等模块，便于测试。
- 最终交付产物仍保持单文件可打开、可演示、可移交。

## 3. 测试分层建议

- 单元测试：覆盖计算函数、格式化函数、状态转换、风险判定。
- DOM 测试：覆盖模块渲染、筛选交互、按钮反馈、联动展示。
- E2E 冒烟测试：覆盖从页面加载到策略演示运行的主流程。

## 5. 推荐测试执行顺序

1. 先完成 `core` 层测试：
   - 净收益计算
   - funding 差计算
   - 风险判定
   - 格式化函数
2. 再完成 `state` 层测试：
   - 筛选状态
   - 策略节点编辑
   - 组合预览生成
   - 运行状态流转
3. 再完成 `render` 层测试：
   - Hero
   - 摘要条
   - 监控矩阵
   - 图表占位区
   - 策略构建器
   - 组合预览
   - 风险面板
   - 日志区
4. 最后完成 E2E 冒烟测试：
   - 页面加载
   - 筛选机会
   - 配置策略
   - 生成组合
   - 启动演示运行
   - 查看日志与状态变化

- `P0` 和 `P1` 所有核心测试通过。
- 至少完成 `P2-01` 响应式布局与 `P2-03` 空态异常态。
- 页面可本地直接打开或通过静态服务器打开。
- 所有执行类按钮均为演示行为，不触发真实交易。
- 页面完整覆盖以下模块：
  - 总览区
  - 机会摘要条
  - 标的筛选栏
  - 永续价差监控矩阵
  - Grafana 图表占位区
  - 图形化策略构建器
  - 套利组合预览卡
  - 风险控制面板
  - 事件日志区
  - 时序数据状态卡
- 核心纯函数具备单元测试。
- 主流程具备 E2E 冒烟覆盖。
- 代码通过 lint，结构清晰，可继续扩展到真实服务接入阶段。

## 8. 建议的首轮迭代顺序

如果需要尽快做出可演示版本，建议首轮开发优先级如下：

1. `P0-01` 到 `P0-05`：先完成基础骨架、数据契约、状态与初始化。
2. `P1-01` 到 `P1-06`：先把“总览 + 筛选 + 监控矩阵 + 联动”打通。
3. `P1-10` 到 `P1-17`：补齐策略构建、组合预览、风控与运行演示。
4. `P1-18` 到 `P1-20`：完成日志、告警与系统联动。
5. `P1-07` 到 `P1-09`：补齐图表占位区与时序状态卡。
6. `P2`：最后做响应式、异常态、可访问性与性能整理。

这样可以先跑通 MVP 的核心主链路，再逐步提高交付质量。
